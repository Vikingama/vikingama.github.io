JavaScript 的事件循环机制

2019.05.13

JS 的并发模型基于“事件循环”，这个模型与其它语言中的模型截然不同。

JS 不具备并行处理任务的能力

当前，JS 的运行环境并不能给予 JS 多线程运行的能力；JS 只能在一个线程上运行，称为主线程。

Web Worker 可以让 JS 具有在独立的子线程上处理费时任务的能力，但它由主线程进行控制，不能访问和更新 DOM。

消息队列/任务队列/事件队列

JS 运行时包含了一个待处理的任务队列（先进先出）；每一个任务都关联着一个用以处理这个任务的函数。

在事件循环期间的某个时刻，从最先进入队列的消息开始处理队列中的消息。为此，这个消息会被移出队列，并作为输入参数调用与之关联的函数。

函数的处理会一直进行到执行栈再次为空为止；然后事件循环将会处理队列中的下一个消息（如果还有的话）。

事件驱动

所谓的事件驱动，就是将一切抽象为事件。IO 操作完成是一个事件，用户点击是一个事件，Ajax 完成是一个事件，一个图片加载完成是一个事件...

“任务队列”中的事件，除了 IO 设备的事件外，还包括一些用户产生的事件；只要指定过回调函数，这些事件发生时就会进入“任务队列”，等待主线程读取。

事件驱动的的实现过程主要靠“事件循环”完成。进程启动后就进入主循环。主进程不停的从事件队列里读取事件。如果事件有关联的处理函数，就执行函数。

主线程运行时，产生堆 Heap 和栈 Stack，栈中的代码调用各种接口，在“任务队列”中加入各种事件；只要栈中的代码执行完毕，主线程就会去读取“任务队列”，依次执行那些事件所对应的回调函数。

JS 通过事件循环机制实现异步

为了协调事件、用户交互、脚本、UI 渲染和网络处理等行为，防止主线程阻塞，JS 引入事件循环机制来管理协调这些事件，实现异步：

  1. 所有任务都在主线程上执行，形成一个“执行栈”；
  2. 把异步任务放到“任务队列”之中，然后继续执行后续的任务；
  3. 一旦“执行栈”中的所有任务执行完毕，系统就会读取“任务队列”。如果这个时候，异步任务已经结束了等待状态，就会从“任务队列”进入“执行栈”执行；
  4. 主线程不断重复上面的第三步。

事件循环中，每一次循环称为 Tick。

MacroTask & MicroTack

任务队列中任务分为：宏任务 MacroTask 和微任务 MicroTack；每次执行栈中的所有宏任务执行完毕后执行所有任务队列中的微任务（如果有的话），然后接着执行所有宏任务（如果有的话），循环往复。
